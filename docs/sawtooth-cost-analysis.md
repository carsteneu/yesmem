# Sawtooth Context Compression - adaptive Context Window - Cost Analysis

**Model:** Claude Opus 4.6 (1M context)
**Data:** 24h proxy logs + Anthropic billing CSV
**Pricing:** 5min Cache TTL

## What Sawtooth Does

Sawtooth keeps a **frozen prefix** of stubbed messages that remains byte-identical across compaction cycles. When context grows past a threshold (~200k tokens), the proxy compacts old messages into stubs — **losslessly**. No information is lost; messages are replaced with structured stubs that preserve key facts, decisions, and tool actions. The original messages remain in the database and can be re-expanded on demand.

This is fundamentally different from Claude Code's native compaction, which uses **lossy summarization** — older messages are replaced with a condensed summary generated by the LLM, permanently losing detail and nuance.

Compaction also **bounds the context size**. Without it, context grows monotonically with each turn until hitting the 1M token ceiling, at which point the session must restart.

## Real-World Data: Typical Day

9 sessions, 1.265 requests. Average context size ~125k tokens.

### Cost Structure

| Component | Tokens | Cost | Share |
|---|---|---|---|
| Cache Read | 164M | $82 | 73% |
| Output | 0.62M | $16 | 14% |
| Cache Write | 2.3M | $14 | 13% |
| **Total** | | **$112** | |

Cache reads dominate at 73% — this is the lever compaction controls.

### What the Proxy Actually Writes

Of the 2.3M write tokens, 96% are incremental writes that occur in **every** scenario — new messages, tool results, assistant responses. Only the COLLAPSE events are proxy-specific:

| Category | Count | Tokens | Cost | Explanation |
|---|---|---|---|---|
| Incremental (new content) | 493 | 1.73M | $11 | Same in all scenarios |
| **COLLAPSE (compaction)** | **9** | **0.61M** | **$4** | **Only proxy-specific cost** |
| Cache hits (no write) | 834 | 0 | $0 | Cached prefix reused |

Write costs are nearly identical across all scenarios because the new suffix written per turn is the same size regardless of whether the cached prefix is 80k or 800k. The difference between scenarios is entirely in **read** costs.

## Key Findings

### 1. Proxy Saves $47/Day vs CC-Native

Average context: ~125k (proxy) vs ~261k (CC-native). This halves read costs ($82 vs $131). Write costs are identical. CC-native compaction never fires for these workloads — no session reaches 835k tokens in a day.

### 2. Compaction Write Overhead is Negligible

9 COLLAPSE events in 24h, costing $4. There is no meaningful write trade-off — the proxy is cheaper on reads AND roughly equal on writes.

### 3. CC-Native = No Compaction

CC-native compaction triggers at ~835k. Since no session reaches that threshold, it behaves identically to no compaction. The proxy's value is **early compaction at 200k**, keeping reads cheap throughout the session.

### 4. /clear@200k Matches the Proxy

Both produce ~164M read tokens at $82. But /clear requires 7 session restarts — each losing all conversation context. The proxy preserves context continuously.

### 5. Cache Reads are 73% of Cost

At $0.50/MTok, reads seem cheap per token — but 164M tokens add up. Keeping average context bounded at ~125k instead of letting it grow to ~261k saves $49/day.

## Recommended Configuration

```yaml
proxy:
  sawtooth_enabled: true
  cache_ttl: "ephemeral"          # 5min TTL
  token_threshold: 200000         # well-validated sweet spot
  token_minimum_threshold: 80000  # @60k saves only $2/day
  cache_keepalive_enabled: true
  cache_keepalive_pings_5m: 6     # bridges gaps up to ~27min
```

## Caveats

1. **Single-day data.** Representative snapshot but workload varies day to day.
2. **CC-native compaction target unknown.** We assume 200k (optimistic). If it compacts to 400k+, the proxy's advantage is larger.
3. **Quality not quantified.** Bounded context (~125k) likely provides better response quality than 800k+.
4. **Counterfactuals are idealized.** They assume perfect cache hits between compactions. Real-world TTL expiry would make alternatives slightly more expensive.
